## When to Invoke

Trigger this skill when:

- **After any UI change** — covers the human-judgment layer that linters and scanners miss
- **When building a new page or route**
- **When adding a form, modal, dialog, or custom interactive component**
- **Before marking a UI task done**

## What This Skill Covers

**This skill focuses on the semantic and interaction layer** — the roughly 40-60% of accessibility issues that automated tools cannot detect.

**Assumed to be handled separately by your toolchain:**
- Static analysis linters (e.g. `eslint-plugin-jsx-a11y`) catch mechanical violations at write time — missing alt attributes, invalid ARIA roles, etc.
- Runtime scanners (e.g. axe-core) catch ~30-40% of WCAG violations automatically
- Color contrast math is handled by design tools and axe

**This skill catches what those tools miss:**
- Labels and alt text that exist but are meaningless
- Button names that don't communicate the action
- Error messages that identify a problem but don't explain how to fix it
- Focus management that technically works but loses the user

## Module 1: Semantic Quality Review

For each interactive or content element in the changed component, evaluate whether the accessibility implementation is *meaningful* — not just syntactically present.

### Images and Icons

Read every `alt` attribute in the changed code. Ask: if a screen reader user heard only this text, would they understand what the image communicates?

**Flag these patterns:**
- `alt=""` on informative images (decorative images are fine with empty alt)
- `alt` values that are filenames: `"hero.png"`, `"icon_check.svg"`
- `alt` values that describe format not content: `"photo"`, `"image"`, `"icon"`
- `alt` values that are keyword lists instead of sentences
- `aria-label` on icon buttons that names the icon, not the action: `"pencil"` should be `"Edit profile"`

**Good:** `alt="Sarah Johnson's profile photo"` / `aria-label="Delete this post"`
**Bad:** `alt="profile"` / `aria-label="trash"`

### Buttons and Links

Read every `<button>`, `<a>`, and element with `role="button"` or `role="link"`. The accessible name must communicate what happens when activated.

**Flag these patterns:**
- Buttons with only an icon and no text or `aria-label`
- Links with text "Read more", "Learn more", "Click here", "Here"
- Buttons labeled by their icon name instead of their action
- Duplicate button names on the same page with different destinations

**Good:** `<button aria-label="Remove Jane from the event">` / `<a>View Sarah's full profile</a>`
**Bad:** `<button><TrashIcon /></button>` (no accessible name) / `<a>Read more</a>`

### Form Labels and Inputs

For every `<input>`, `<select>`, `<textarea>` — confirm it has a visible label associated via `htmlFor`/`id` or `aria-labelledby`. Then evaluate whether the label is descriptive.

**Flag these patterns:**
- Inputs with `placeholder` as the only label (disappears on focus)
- Labels that only state the field type: "Text", "Input", "Field"
- Labels that don't communicate format requirements when format matters (phone, date, postal code)
- Password fields with no description of requirements

**Good:** `<label>Email address (we'll send a confirmation)</label>`
**Bad:** `<label>Email</label>` paired with `placeholder="Enter your email"`

### Error Messages

When a form component has validation or error states, check that error messages:
1. Identify what went wrong (not just "Invalid")
2. Tell the user how to fix it
3. Are associated with the relevant input via `aria-describedby`

**Good:** `"Password must be at least 8 characters and include one number"`
**Bad:** `"Invalid password"` or `"Error"`

### Dynamic Content

If the component updates content without a page reload (loading states, success messages, real-time updates), confirm it uses `aria-live="polite"` or `role="status"` so screen readers announce the change. Urgent alerts (errors, warnings) should use `aria-live="assertive"` or `role="alert"`.

## Module 2: Interaction Flow Check

Evaluate whether the component works correctly for keyboard-only users.

### Tab Order and Focus Visibility

Review the component's DOM order and any `tabIndex` usage.

**Check:**
- Interactive elements appear in a logical tab sequence (matches visual reading order)
- No `tabIndex` values greater than 0 (these hijack the natural order)
- No `:focus { outline: none }` or `outline: 0` without a visible replacement
- Focus indicators meet WCAG 2.2: at least 2px, 3:1 contrast against surrounding color

**Flag immediately:** Any CSS that removes focus indicators without replacing them.

### Modal and Dialog Focus Management

When a modal, drawer, sheet, or dialog opens:

1. **Focus moves in** — first interactive element or modal heading receives focus on open
2. **Focus is trapped** — Tab cycles only within the modal; nothing behind it is reachable
3. **Focus returns** — when modal closes, focus goes back to the element that opened it

Check that the component implements all three. If using a component library that handles focus trapping (e.g. Radix, Headless UI, Mantine), confirm the default behavior isn't being overridden.

### Button and Link Keyboard Behavior

- `<button>` elements: activate with both Enter and Space
- `<a>` elements with `href`: activate with Enter
- Custom interactive elements using `role="button"` on a `<div>` or `<span>`: must have `tabIndex="0"` and keyboard event handlers for Enter and Space

**Flag:** Any `onClick` handler on a non-interactive element that doesn't also handle `onKeyDown` for Enter/Space.

### Skip Navigation

For new full pages (not components): confirm a skip-to-main-content link is present, or that the existing layout provides one. Users shouldn't Tab through the full navigation on every page visit.

## How to Report Findings

After reviewing, structure output as:

```
## Accessibility Review: [Component/Page Name]

### Passes
- [what was checked and confirmed good]

### Issues Found
- [WCAG criterion] [Severity: Critical/High/Medium/Low]: [specific issue]
  Fix: [concrete remediation]

### Automated Coverage Note
- Recommend running axe-core (via checkA11y or similar) in E2E tests
  for this page to catch mechanical violations automatically.
```

### Severity Definitions

| Severity | Meaning |
|----------|---------|
| **Critical** | Blocks a user from accessing content or completing a task |
| **High** | Makes a task significantly harder; requires workarounds |
| **Medium** | Creates notable difficulty; some users can work around it |
| **Low** | Minor friction; doesn't prevent completion |

## Adding Automated Accessibility Checks to E2E Tests

When building a new page or route, add an axe-core accessibility check to its E2E spec. Most test frameworks have axe integrations:

**Playwright (with @axe-core/playwright):**
```typescript
import { test, expect } from '@playwright/test';
import AxeBuilder from '@axe-core/playwright';

test('page has no accessibility violations', async ({ page }) => {
  await page.goto('/your-page');
  await page.waitForLoadState('networkidle');
  const results = await new AxeBuilder({ page }).analyze();
  expect(results.violations).toEqual([]);
});
```

**Cypress (with cypress-axe):**
```typescript
it('has no accessibility violations', () => {
  cy.visit('/your-page');
  cy.injectAxe();
  cy.checkA11y();
});
```

For pages with known outstanding issues you're not fixing right now, log violations without failing the test so you track the debt without blocking CI.

## Anti-Patterns

```
"axe didn't flag anything so it's accessible"
  Axe catches ~30-40% of issues. Semantic quality is your job.

"it has an aria-label so it's fine"
  Does the label communicate meaning? "icon" is not meaning.

"the label is there"
  Is it visible? Is it associated via htmlFor/id? Does it describe the field?

Skipping the interaction check for "simple" components
  A simple icon button with no accessible name blocks keyboard users entirely.
```
